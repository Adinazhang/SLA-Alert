一、背景与目标#

1.1 背景

当前 DTC 跨境物流轨迹接入后,系统通过 节点匹配规则(rule_node) 将物流商原始轨迹自动归入系统节点,并基于 预警监控规则(node_monitor_rule) 对超时节点触发时效预警。两个问题:

  1. 节点覆盖不全:现有匹配节点集中在「物流已收货 → 妥投」阶段,订单预报、出库交接、进口清关三个关键节点在配置层未启用或缺失,导致时效监控无法覆盖全链路。
  2. 匹配规则容错低:当前只有单层关键词,遇到物流商文案变化就会「漏匹配」,漏匹配的轨迹只是静默落地,没有人知道出了问题,时效预警也无从触发。
  3. 预警维度单一:现仅支持「节点超时」一种预警,无法对原始轨迹中的关键词(如「破损」「扣留」「无法投递」)做分色级即时告警,运营难以在轨迹层面提前介入。

1.2 目标

  • 节点全链路覆盖:补齐订单预报、出库交接、进口清关三个节点。
  • 双层兜底匹配:一级关键词为主,二级关键词兜底,双层均未命中则显性报警,从「静默漏匹配」变为「可见可处理」。
  • 关键词预警 + 列表交互增强:在预警监控规则中新增「关键词预警配置」(按区间配置 + 颜色分级 + 是否处理),预警列表支持顶部色级卡片快捷筛选与批量处理(按所选 / 按条件)。
  • 节点时效阈值精细化:路由节点时效单位由「天」改为「h」,新增 TOTAL 全链路总时长合计,预警触发条件公式化(实际时长 > 参考阈值 ×(1 + 预警触发%))。
  • 新增数据看板:节点时效预警菜单下新增「数据看板」Tab,统计工单总数 / 已处理 / 平均响应时长 / 单均处理时长 4 张卡片,以及 SLA 达标率、平均妥投天数、异常订单占比等业务指标。

1.3 非目标 (Out of Scope)

  • 不涉及物流商接入方式改动(Webhook/拉取逻辑不变)。
  • 不重算历史轨迹,仅对上线后新增轨迹生效。

二级关键词一级未命中时的兜底词

三、需求一 · 节点扩展 + 一二级关键词匹配#

3.1 节点清单(配置顺序)

订单预报(101) → 出库交接(103) → 物流已收货(200) → 离开始发国(202)
  → 到达目的国(203) → 进口清关(新增) → 末端提取(204)
  → 运输途中(205) → 妥投(206)
节点处理方式说明
订单预报配置启用枚举已存在(101),运营在节点匹配规则页配置关键词即可
出库交接配置启用枚举已存在(103),同上
进口清关新增枚举 + 配置启用建议编码 208(与研发确认,避开已占用段位)

3.2 匹配规则(核心)

对每条原始轨迹,按系统节点顺序执行:

1. 用 match_keywords(一级)匹配
     └─ 命中 → 标记 match_level=1,落为对应节点轨迹
2. 一级未命中 → 用 second_match_keywords(二级)匹配
     └─ 命中 → 标记 match_level=2,落为对应节点轨迹(前端加"二级"角标)
3. 二级仍未命中 → 标记 is_abnormal=1,进入异常轨迹池
  • 保留现有 exclude_keywords、necessary 行为不变。
  • 现有匹配顺序规则(按节点顺序匹配、已匹配轨迹不参与后续)保持不变。

3.3 异常轨迹 → 时效预警

  • 节点匹配流程结束后,扫描该物流单下 is_abnormal=1 的原始轨迹。
  • 如存在异常轨迹,生成一条 alert_type=2 的 node_monitor_alert,进入现有预警列表。
  • 去重策略:tracking_number_id + 原始轨迹内容 MD5 去重,同一内容不重复告警。
  • 关闭策略:运营在预警详情页回补关键词并触发「重新匹配」,若 is_abnormal 全部变 0,预警自动流转至「已处理」。

四、需求二 · 关键词预警 + 预警列表交互增强#

本章基于 index.html 原型 与《页面交互说明》同步细化。三条主线:

1) 预警监控规则页 — 新增「关键词预警配置」(按区间 + 关键词 + 颜色 + 是否处理);同时路由节点时效 & 阈值设置调整时效单位为 h、引入 TOTAL 合计、预警触发公式化。

2) 节点时效预警 · 明细列表 — 顶部统计卡片快捷筛选、搜索条件扩展、类型配色、批量处理(按所选 / 按条件)、单条处理弹框信息扩充(占比 / 轨迹高亮 / 多条件顺序告警)。

3) 节点时效预警 · 数据看板 — 新增四张统计卡片 + SLA 达标率 / 平均妥投天数 / 异常订单占比 等业务指标,辅助运营管控全链路时效。

4.1 关键词预警规则配置(预警监控规则页)

4.1.1 表结构(每条规则一个区间)

每条物流商 + 渠道的预警监控规则,新增「关键词预警配置」区,按行配置。每行字段:

字段说明
开始节点监控区间起点(从系统 9 节点中单选)
结束节点监控区间终点(从系统 9 节点中单选)
关键词多 tag 输入(回车/逗号添加),不区分大小写、忽略首尾空格
预警是否处理是 / 否,默认「是」。仅作用于命中后生成的预警工单状态:= 是 → 工单走 待处理 → 处理中 → 已处理 状态机;= 否 → 工单直接落「无需处理」状态(系统自动归档,不入运营处理队列)
预警颜色黄 / 红 / 黑,必选(无论「预警是否处理」选择是 / 否,颜色字段都需要选择;该颜色会写入命中后生成的预警工单,用于色级筛选与展示)
操作删除整行

评估顺序(单条物流单):先逐条原始轨迹查关键词预警 → 命中即按本行配置颜色生成预警工单 → 均未命中再走下方「路由节点时效阈值」判定。

4.1.2 关键交互(原型)

  • 预警颜色色块组始终展示并可选;无论「预警是否处理」选择是 / 否,颜色都必选。未选颜色时保存失败,色块组旁红字提示「请选择」。
  • 「预警是否处理」与「预警颜色」彼此独立:切换是 ↔ 否 不会清空已选颜色,色块组也不隐藏;两个字段分别决定工单的处理状态与展示颜色。
  • 预警是否处理 = 是 → 命中后生成的工单走 待处理 → 处理中 → 已处理 状态机,可在明细列表批量 / 单条处理。
  • 预警是否处理 = 否 → 命中后仍按所选预警颜色生成工单,工单直接落「无需处理」状态(处理状态字段灰字 #909399,操作列仅「查看」),适用于「Pickup attempted / 投递改约」等仅作记录、无需运营介入的提示性文案。
  • 关键词 tag 颜色 = 预警颜色:行内已添加的关键词 tag 自动随当前预警颜色着色(黄 / 红 / 黑),不受「预警是否处理」影响。

4.1.3 字段变更(DB)

-- 关键词预警规则主表
CREATE TABLE alert_keyword_rule (

4.2 路由节点时效 & 阈值设置(同页)

原「预警监控规则」编辑页内的「路由节点时效 & 阈值设置」区,本次调整:

  • 时效单位调整:原以「天」为单位的标准时效/参考阈值字段,统一改为 h(小时),粒度更细,支撑短链路渠道精细化预警。
  • 新增 TOTAL 行:依据当前已配置的路由节点,实时合计 SUM(标准时效) 与 SUM(参考阈值),展示开始 → 结束全链路总时长,辅助评估整体 SLA 是否合理。节点行新增/删除/编辑后,TOTAL 自动重算。
  • 预警触发公式: 
    当 实际时长 > 参考阈值(h) × (1 + 预警触发%) 时触发节点超时预警
按本节点所属预警监控规则的颜色生成 alert_type=1 工单
  • 评估顺序:同一物流单的同一区间,先关键词预警(§4.1) → 再节点时效预警(本节);关键词命中即跳过本区间节点时效判定,避免双重告警。

4.5 明细列表 · 类型字段配色与处理状态

字段值颜色规范

关键词触发绿色徽标 bg-#f0f9eb / text-#67c23a / border-#b3e19d

处理状态机:

  • alert_type=1 节点超时 → 待处理 → 处理中 → 已处理,操作列展示「处理」入口。

4.6 明细列表 · 批量处理

明细列表支持两种批量模式:

  • 按所选批量:勾选目标行的复选框后点击顶部「批量处理」,弹框头部展示「已选 N 条」。
  • 按条件批量:不勾选任何行直接点「批量处理」,系统以当前筛选条件命中的全部工单作为批量目标,弹框头部展示「按条件批量处理 N 条」并附筛选条件摘要。

批量处理弹框内容仅展示「处理操作」(剥离单条预警弹框中的订单信息/监控节点详情/轨迹时间轴等),共用以下结构:

  • 单选「关联到实际已到达节点」(默认选中,带「实际已到达节点」下拉,从 9 个系统节点单选)
  • 单选「仅标记为正常(不关联节点)」
  • 文本域「处理备注」(非必填)
  • 底部「关闭 / 提交处理」;关闭 / ESC / 点遮罩 均可关闭弹框

限制:批量处理仅适用于 need_handle=1 的工单(节点超时 / 关键词触发-需处理);need_handle=0 的「无需处理」工单不参与批量(行复选框禁用)。

4.7 明细列表 · 单条处理弹框

单条预警的「处理」入口弹框结构:

  • 头部:预警编号 + 色级徽标
  • 主体(自上而下):
    • 订单基础信息:订单号、物流商 / 渠道、目的国、客户等
    • 监控节点详情:展示触发的节点、标准时效(h)、参考阈值(h)、实际时长(h)、超时用时(h)、占比;
      • 占比计算:占比 = 超时用时 / 实际用时,以 % 单位展示;若计算结果有小数,保留两位小数
    • 物流轨迹(系统节点轨迹):若该轨迹命中预警条件(关键词命中 或 节点超时),整行高亮标识(色块/边框,与预警颜色一致)
    • 原始轨迹(物流商原文):同样命中预警条件的行需高亮标识,便于运营快速定位
    • 处理操作:同 §4.6 弹框(关联节点 / 仅标记正常)
    • 处理备注:文本域,非必填
  • 底部:关闭 / 保存(中间态保存为「处理中」) / 提交处理(关单为「已处理」)

多条件预警告警顺序:同一渠道下若多条监控节点同时满足触发条件,告警工单按命中时间先后顺序逐条产出(先触发的先告警);前一条工单进入「已处理」后,后一条工单才会进入「待处理」队列,避免一次性堆积无法分辨主因。

4.8 数据看板 Tab(节点时效预警菜单)

查询条件与统计卡片联动:数据看板顶部查询模块包含仓库、物流商 / 渠道、国家、日期四类条件,以及「查询 / 重置」按钮。当前页面的 4 张统计卡片必须根据查询条件动态展示:点击「查询」后,工单总数、已处理、平均响应时长、单均处理时长均按当前条件命中的工单范围重新计算;点击「重置」后,查询条件恢复默认值,统计卡片同步刷新为默认查询范围的数据。

卡片口径说明

4.8.2 业务指标

指标公式举例
SLA 达标率 阈值范围内订单数 / 总订单量 全程时效阈值 7 天,总订单 1000,7 天内达标 800,达标率 = 800 / 1000 = 80%
平均妥投天数 Σ(妥投时间 − 发货时间) / 已妥投订单数 A 5 天 / B 6 天 / C 4 天,平均 = (5 + 6 + 4) / 3 = 5 天

  • 筛选要求:
    • 仓库:支持多选
    • 物流商 / 渠道:单选;必须选定物流商 / 渠道后列表才展示节点时效指标数据,默认空态提示「请选择物流商 / 渠道后查看节点时效指标」

七、附:运营使用指引(随上线一并产出)#

  1. 一级关键词:写「物流商最主流、最稳定的文案」,如「Out for delivery」、「已签收」。
  2. 二级关键词:写「变体、小语种、同义文案」,如「Delivered to recipient」、「已完成投递」。
  3. 异常轨迹回补流程:
    • 时效预警列表 → 筛选「轨迹未命中」
    • 点开详情 → 复制原始文案
    • 点「回补关键词」→ 在对应节点的二级栏位追加
    • 保存 → 触发重算 → 预警自动关闭
  4. 关键词预警配置:
    • 在「预警监控规则」编辑页 → 「关键词预警配置」区,按区间(开始→结束节点)按行配置;
    • 无论「预警是否处理」选择是 / 否,都必须选预警颜色(黄/红/黑);
    • 「预警是否处理 = 否」适用于「Pickup attempted / 投递改约」等仅作记录、无需处理的提示性文案;命中后仍按所选颜色生成预警工单,工单状态直接落「无需处理」;
    • 关键词 tag 颜色会自动随所选预警颜色高亮,便于核对一致性。
  5. 明细列表快捷筛选:点击顶部色级卡片(黄/红/黑)即按紧急度筛选,再次点击取消;两张白底卡片仅用于数据展示,不可筛选。
  6. 批量处理:勾选多条「节点超时」或「关键词触发-需处理」工单 → 点「批量处理」→ 选择「关联节点 / 标记为正常」并填备注,一键关单。need_handle=0 的「关键词触发-无需处理」工单不参与批量。
